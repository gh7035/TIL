# LoadBalancer란?

일단 말 그대로 **Balancer**. 균형을 유지 시키는 것이다.

서버에 들어오는 트레픽을 분산시켜주는 또 다른 서버이다.

보통 **스케일 아웃** 일떄 사용하는데 우선 스케일 아웃부터 알아보자면

- 스케일 업 : i4 컴퓨터 를 i6, i7으로 사양을 올리는 것
- 스케일 아웃 : i4의 컴퓨터를 여러대 두는 것

스케일 업을 하면 비용은 많이 들지만, 관리가 용이하지만 장애 대응이 어렵고,
스케일 다운시에는 비용은 스케일 업보다는 적게 들지만 관리 편의성이 떨어지고, 컴퓨터를 여러대 둘수록 운영비용이 더 들어가기도 하지만, 서버가 분산되어있기 떄문에 장애 대응이 스케일 업보다 비교적 쉽다.

## LoadBalancer 알고리즘

크게 **정적 로드 밸런싱**, **동적 로드 밸런싱**이 존재한다.

### 정적 로드 밸런싱
1. 라운드 로빈 방식
클라이언트의 요청을 서버들에게 순차적으로 분배하는 방식이다. 클라이언트의 요청을 순서대로 분배하기 때문에 서버들이 동일한 스펙을 가지고 있고, 서버와의 연결이 오래 지속되지 않아도 되는 경우 활용하기 적합하다. 하지만 성능이 낮은 서버가 처리 속도를 따라잡지 못해 먼저 뻗어버릴 수 있다. \
A, B, C의 서버를 가지고 있을 경우 A > B > C > A 순서대로 순회한다.


2. 가중치 기반 라운드 로빈 방식
각각의 서버마다 가중치를 설정하고, 가중치가 높은 서버에 클라이언트의 요청을 먼저 배분한다. 여러 서버가 같은 사양이 아니고, 특정 서버의 스펙이 좋은 경우 해당 서버의 가중치를 높게 매겨 트래픽 처리량을 늘릴 수 있다. 하지만 가중치가 높은 서버가 우선적으로 요청을 담당하다가 갑자기 무거운 요청을 받게되면 장애가 생길 수 있다.

3. ip 해시 방식
클라이언트의 ip를 해시화 해서 항상 동일한 서버로 요청을 보낸다. 보통 한 세션이 오래 남아 있어야 하는 경우에 사용한다. 하지만 한 '단체'에서 같은 ip를 사용시 부하가 걸릴수 있는 단점이 있다.

### 동적 로드 밸런싱
1. 최소 연결 방식
현재 클라이언트의 요청을 가장 적게 처리한 서버를 찾아서 새 요청을 보내게 한다.

2. 최소 응답 시간 방식
서버의 연결된 요청 수와 응답 시간을 고려하여 가장 빠른 응답이 가능한 서버를 계산해서 클라이언트 요청을 할당한다.

3. 리소스 기반 방식
각 서버에 별도의 측정 프로그램을 구비하여 서버의 리소스가 가장 여유있는 서버에게 클라이언트 요청을 할당한다.

## 정적 로드 vs 동적 로드

각각의 장단점이 있는데, 정적 로드 밸런싱은 서버의 상태는 고려하지 않고 규칙에 따라 작동하기에 구현 난이도가 쉽고, 오버헤드가 적지만, 동적 로드 밸런싱은 서버의 상태를 항상 고려하기 때문에 자원의 이용 효율을 극대화 할수 있지만, 오버헤드가 커서 계산 부하가 발생할 수 있다.

#### 결론
로드 밸런싱 사용시에는 트레이드 오브를 잘 해서 방식을 잘 결정해야한다.
